# 0.2. Part별 바이브 코딩 프롬프트 가이드

바이브 코딩은 **“무슨 느낌/결과를 원하는지”**를 중심으로 AI에게 요청하는 방식이다.  
아래는 **Part(단계)별**로 나눈 **프롬프트 구조**와 **실제로 복사해 쓸 수 있는 예시**다.

---

## 코드 복사 사용법

- 각 예시는 **코드 블록(```)** 안에 들어 있다. 블록 **전체를 복사**해서 AI 채팅창에 붙여넣으면 된다.
- `[ ]` 안 문구나 하이픈(`-`) 아래 항목을 **프로젝트/기능에 맞게만 바꿔서** 쓰면 된다.
- "짧게 쓸 때"용 한 줄 프롬프트는 문서 맨 아래 **§ 한 줄 프롬프트 (복사용)** 에 모아 두었다.

---

## 1. 바이브 코딩 프롬프트 구조

한 줄 요약: **역할 + 상황/목표 + 제약(선택) + 원하는 출력 형태**

| 요소 | 설명 | 예시 |
|------|------|------|
| **역할** | AI가 어떤 관점으로 답할지 | "리팩터링 관점에서", "프론트엔드만" |
| **상황/목표** | 지금 무엇을 만들고 싶은지, 어떤 문제를 풀고 싶은지 | "로그인 폼 추가", "이 API 호출이 느려" |
| **제약** | 스택, 스타일, 금지 사항 | "React만", "인라인 스타일 금지", "기존 API 유지" |
| **출력 형태** | 코드만 / 설명 포함 / 단계별 등 | "코드만", "단계별로 설명하면서" |

---

## 2. Part 1: 기획·설계

**목적:** 요구사항을 구조·플로우·데이터 모델로 정리할 때 사용.

### 2.1 구조

- **역할:** 기획/설계 조언자  
- **상황:** 기능 범위, 화면/API 구성, 데이터 구조  
- **제약:** 기술 스택, 레거시 연동 여부  
- **출력:** 목록/플로우/ER 개요, 선택지와 트레이드오프

### 2.2 복사해서 쓸 수 있는 프롬프트

```
[기획·설계] 우리 서비스에 "주문 취소 후 부분 환불" 기능을 넣으려고 해.
- 역할: 기획/설계 관점에서 조언해줘.
- 상황: 주문은 이미 결제 완료 상태고, 취소 시 일부 상품만 환불하는 케이스를 지원해야 함.
- 제약: 기존 주문/결제 테이블 스키마는 최소한으로만 변경하고 싶음.
- 출력: (1) 필요한 API/화면 목록, (2) 상태 플로우(주문→취소 요청→환불 처리), (3) DB 변경 포인트만 간단히 정리해줘.
```

```
[기획·설계] 소규모 팀용 "일일 스탠드업 봇"을 슬랙으로 만들 예정이야.
- 역할: 기능 스코프와 설계 관점으로 정리해줘.
- 상황: 매일 아침 9시에 슬랙 채널에 질문을 보내고, 멤버가 스레드로 답하면 요약 메시지를 한 번에 올리는 흐름.
- 제약: 무료 슬랙 앱 범위, 서버는 1개(Node 또는 Python).
- 출력: (1) 필요한 슬랙 이벤트/API 목록, (2) 봇–서버 간 데이터 흐름 한 줄 요약, (3) MVP에 꼭 넣을 기능 3개만 골라줘.
```

## 3. Part 2: 기능 구현 (코드 작성)

**목적:** 구체적인 기능/API/로직을 코드로 받을 때 사용.

### 3.1 구조

- **역할:** 백엔드/프론트/풀스택 등 구현 담당  
- **상황:** 어떤 기능을, 어떤 파일/컴포넌트에 넣을지  
- **제약:** 언어, 프레임워크, 기존 코드 스타일  
- **출력:** 코드 + 필요한 경우 짧은 설명

### 3.2 복사해서 쓸 수 있는 프롬프트

```
[기능 구현] 로그인 폼을 하나 추가해줘.
- 역할: 프론트엔드 구현. React + 기존 디자인 시스템(버튼/인풋) 사용.
- 상황: 이메일 + 비밀번호 입력, 제출 시 `/api/auth/login` POST 호출, 성공 시 `/dashboard`로 이동.
- 제약: 기존 프로젝트의 `Form`, `Input`, `Button` 컴포넌트만 써줘. 새 라이브러리 추가하지 마.
- 출력: 로그인 전용 페이지 컴포넌트 코드만 작성해줘. (에러 메시지 표시 포함)
```

```
[기능 구현] 결제 완료 후 "주문 상세" API를 하나 만들어줘.
- 역할: 백엔드 API 구현. 우리 프로젝트는 FastAPI + SQLAlchemy 사용 중.
- 상황: GET /orders/{order_id} 로 주문 한 건 조회. 주문 정보 + 주문된 상품 목록 + 배송지 반환.
- 제약: 이미 있는 Order, OrderItem, Address 모델만 사용. N+1 쿼리 없이 한 번에 조회해줘.
- 출력: 라우트 함수와 필요한 서비스/쿼리 코드만. 응답 스키마(Pydantic) 포함.
```

```
[기능 구현] 사용자가 드래그로 순서를 바꿀 수 있는 "할 일 목록"을 구현해줘.
- 역할: 프론트엔드. React 사용.
- 상황: 리스트 아이템을 드래그 앤 드롭으로 순서 변경, 변경된 순서를 PATCH /todos/reorder 로 저장.
- 제약: @dnd-kit/core 또는 HTML5 DnD 중 하나만 사용. 기존 Todo 타입(id, title, order) 유지.
- 출력: 목록 컴포넌트와 reorder API 호출 부분 코드. 타입 정의 포함.
```

## 4. Part 3: UI·스타일 (감각/바이브)

**목적:** “어떤 분위기/스타일로 보이게 할지”를 전달할 때 사용.

### 4.1 구조

- **역할:** UI/디자인 구현  
- **상황:** 어떤 화면/컴포넌트를 어떤 톤으로  
- **제약:** 다크/라이트, 반응형, 접근성, 사용 CSS 방법  
- **출력:** 컴포넌트 + 스타일 코드

### 4.2 복사해서 쓸 수 있는 프롬프트

```
[UI·스타일] 랜딩 페이지 상단 히어로 섹션을 만들어줘.
- 역할: UI 구현. 감각적으로.
- 상황: 대시보드형 SaaS 제품 소개. 한 줄 캐치프레이즈 + CTA 버튼 1개.
- 제약: 다크 모드 기준, 그라데이션 배경, 글래스모피즘 카드 느낌. Tailwind 사용. 모바일 우선 반응형.
- 출력: Hero 섹션 HTML/JSX + Tailwind 클래스만. 폰트는 기존 프로젝트 것 유지.
```

```
[UI·스타일] 설정 페이지의 "알림 설정" 블록을 정리해줘.
- 역할: UI. 우리 앱은 미니멀하고 여백이 많은 스타일이야.
- 상황: 알림 on/off 스위치 3개(이메일, 푸시, 슬랙). 각 항목에 짧은 설명 라벨.
- 제약: CSS Modules 또는 styled-components. 기존 Switch, Label 컴포넌트 재사용. 색은 기존 테마 변수만 사용.
- 출력: 설정 블록 컴포넌트 코드. 레이아웃과 스타일 포함.
```

## 5. Part 4: 리팩터링·정리

**목적:** 구조 개선, 중복 제거, 패턴 통일할 때 사용.

### 5.1 구조

- **역할:** 리팩터링/코드 품질  
- **상황:** 어떤 파일/기능을 어떻게 개선할지  
- **제약:** 동작 변경 없음, 테스트 유지, 사용할 패턴  
- **출력:** 변경된 코드 + 변경 요약

### 5.2 복사해서 쓸 수 있는 프롬프트

```
[리팩터링] 이 컴포넌트의 비즈니스 로직을 훅으로 분리해줘.
- 역할: 리팩터링. 동작은 그대로 유지.
- 상황: [파일 경로 또는 코드 붙여넣기] — 폼 제출, 유효성 검사, API 호출이 한 컴포넌트에 다 있음.
- 제약: useFormSubmit 같은 커스텀 훅으로 빼고, 컴포넌트는 UI만 담당하게. 기존 props/인터페이스 유지.
- 출력: 훅 파일 + 수정된 컴포넌트 코드. 변경된 부분만 짧게 요약해줘.
```

```
[리팩터링] 이 디렉터리 안 API 호출을 한 곳으로 모아줘.
- 역할: 리팩터링. 기존 API 스펙과 응답 타입은 그대로.
- 상황: [폴더 경로] 에서 fetch/axios 호출이 여러 파일에 흩어져 있음.
- 제약: api/client.ts 같은 단일 진입점에서 함수별로 export. 에러 처리 방식은 기존과 동일하게.
- 출력: 새 API 모듈 코드 + 기존 파일에서 어떻게 호출을 대체할지 한 줄씩만 적어줘.
```

## 6. Part 5: 테스트·디버깅

**목적:** 테스트 작성, 버그 원인 분석, 로그/에러 개선할 때 사용.

### 6.1 구조

- **역할:** 테스트/디버깅  
- **상황:** 어떤 것을 검증하거나, 어떤 증상인지  
- **제약:** 테스트 러너, mocking 방식, 로그 포맷  
- **출력:** 테스트 코드 또는 원인 분석 + 수정 제안

### 6.2 복사해서 쓸 수 있는 프롬프트

```
[테스트] 이 함수에 대한 단위 테스트를 작성해줘.
- 역할: 테스트 작성. 우리 프로젝트는 Vitest + React Testing Library 사용.
- 상황: [함수/컴포넌트 경로 또는 코드] — 입력에 따라 포맷된 문자열을 반환하는 유틸 함수.
- 제약: 경계값(빈 문자열, null, 긴 문자열) 케이스 포함. 외부 의존성은 mock.
- 출력: 테스트 파일 하나. describe/it 구조로, 케이스별로 주석 한 줄씩.
```

```
[디버깅] 이 API 호출이 프로덕션에서 가끔 타임아웃이 나. 원인 후보와 수정안을 제안해줘.
- 역할: 디버깅/성능 개선.
- 상황: [해당 API 호출 코드 또는 스니펫] — GET /reports/summary, 타임아웃 5초.
- 제약: 서버 코드는 당장 수정 어렵고, 클라이언트에서 할 수 있는 것 위주로. 로깅 추가 가능.
- 출력: (1) 가능한 원인 3가지, (2) 클라이언트에서 적용할 수 있는 수정(재시도/캐시/로깅) 코드, (3) 서버 측에서 나중에 보면 좋은 포인트 한 줄씩.
```

## 7. Part 6: 문서화

**목적:** README, API 문서, 주석, 온보딩용 설명을 작성할 때 사용.

### 7.1 구조

- **역할:** 문서 작성  
- **상황:** 어떤 대상을, 누구를 위해 설명할지  
- **제약:** 포맷(Markdown/OpenAPI 등), 언어, 분량  
- **출력:** 문서 초안 또는 주석 블록

### 7.2 복사해서 쓸 수 있는 프롬프트

```
[문서화] 이 프로젝트의 로컬 실행 방법을 README에 추가해줘.
- 역할: 문서 작성. 새로 합류하는 개발자가 따라할 수 있게.
- 상황: Node 20 + pnpm, 환경 변수 .env.example 필요, DB는 Docker Compose로 띄움.
- 제약: 기존 README 구조 유지. "개발 환경 설정" 섹션 하나만 추가. 명령어는 복사해서 쓸 수 있게.
- 출력: 추가할 Markdown 블록만. 코드 블록 포함.
```

```
[문서화] 방금 만든 주문 API 3개에 대한 OpenAPI 스펙 요약을 만들어줘.
- 역할: API 문서화.
- 상황: POST /orders, GET /orders/{id}, PATCH /orders/{id}/cancel. 요청/응답 필드와 상태 코드만 정리.
- 제약: OpenAPI 3.0 형식. 인증은 Bearer 토큰이라고만 명시. 예시 값 하나씩 포함.
- 출력: paths 와 schemas 만 있는 YAML 스니펫.
```

## 8. 프롬프트 템플릿 (빈 칸 채워서 복사)

아래는 Part 공통으로 **빈 칸만 채워서** 복사해 쓸 수 있는 템플릿이다.

```
[Part: ________ ] ________(하고 싶은 일) 해줘.
- 역할: ________(AI의 관점/역할).
- 상황: ________(현재 맥락, 대상 파일/기능, 조건).
- 제약: ________(스택, 스타일, 금지 사항).
- 출력: ________(코드만 / 설명 포함 / 단계별 등).
```

**Part 이름 예시:** 기획·설계 / 기능 구현 / UI·스타일 / 리팩터링 / 테스트·디버깅 / 문서화

---

## 9. 사용 팁

1. **한 번에 한 Part·한 목적만** 주는 편이 답이 안정적이다.  
2. **파일 경로나 코드 조각**을 붙여 넣으면 "이거 기준으로" 요청하기 좋다.  
3. **출력 형태**를 명시하면(코드만 / 요약만 / 단계별) 불필요한 설명을 줄일 수 있다.  
4. **제약**에 "기존 코드/스타일 유지"를 넣으면 프로젝트 일관성이 유지된다.

이 가이드의 각 예시 블록은 그대로 복사한 뒤, [ ] 안 내용과 하이픈 아래 항목만 프로젝트에 맞게 바꿔서 쓰면 된다.

---

## § 한 줄 프롬프트 (복사용)

바쁠 때는 아래처럼 **한 줄로** 복사해서 쓴 뒤, 필요하면 다음 메시지로 "제약: React만", "출력: 코드만"처럼 보완해도 된다.

```
[기획·설계] [기능명] 기능 스코프랑 필요한 API/화면 목록만 짧게 정리해줘.
```

```
[기능 구현] [기능 설명] 구현해줘. [프레임워크/언어]. 제약: [한 줄]. 출력: 코드만.
```

```
[UI·스타일] [화면/컴포넌트] 이렇게 만들어줘: [감각 키워드, 예: 미니멀/다크/글래스]. [Tailwind/CSS]. 출력: 코드만.
```

```
[리팩터링] [파일/폴더] 이 부분 리팩터링해줘. 동작 변경 없이. [원하는 형태 한 줄]. 출력: 변경된 코드 + 요약.
```

```
[테스트] [함수/컴포넌트 경로] 이거 단위 테스트 작성해줘. [Vitest/jest 등]. 경계값 케이스 포함. 출력: 테스트 코드만.
```

```
[디버깅] [증상 한 줄]. [관련 코드/경로]. 원인 후보랑 수정안 제안해줘.

# 부록: 유니버셜 활용 메뉴얼 (2026년 3월 기준)

이 가이드의 **역할·상황·제약·출력** 구조와 Part별 프롬프트는 **Claude Code**뿐 아니라 아래 플랫폼에서도 그대로 활용할 수 있다. 각 도구마다 입력 위치·단축키·추가 팁만 다르다.

---

## 공통 원칙 (모든 플랫폼)

| 원칙 | 설명 |
|------|------|
| **구조 통일** | `[Part] + 역할 + 상황 + 제약 + 출력` 형식을 유지하면 어떤 AI 코딩 도구에서도 결과가 안정적이다. |
| **한 번에 한 목적** | 한 메시지에 한 Part·한 기능만 요청한다. |
| **출력 형태 명시** | "코드만", "단계별 설명", "요약만" 등을 적어 두면 불필요한 설명을 줄일 수 있다. |
| **제약으로 스코프 고정** | "기존 컴포넌트만", "이 API 유지", "테스트 깨지지 않게" 등을 넣어 범위를 제한한다. |

---

## 플랫폼별 활용 요약

### 1. Cursor AI (Cursor IDE)

| 항목 | 내용 |
|------|------|
| **입력 위치** | **Chat**: 사이드 채팅창. **Composer**: `⌘I`(Mac) / `Ctrl+I`(Win·Linux) — 멀티파일 편집·에이전트 작업에 적합. |
| **에이전트 모드** | Composer에서 `⌘.` 로 에이전트 ON. 코드베이스 검색·파일 생성·터미널 실행 등 자동 수행(최대 25 툴 호출). Claude 모델 지원. |
| **컨텍스트** | `@` 입력 후 파일(`#파일명`), 폴더, 규칙 등 첨부. 에이전트 시 `@Recommended` 로 자동 관련 컨텍스트 수집. |
| **활용 팁** | 기획·설계는 Chat으로 요약 받고, 기능 구현·리팩터링은 Composer + 에이전트로 "코드만" 출력 요청. 변경 후 diff 확인·체크포인트로 되돌리기. |

**복사용 한 줄:**  
`[Part] [내용]. 역할: [한 줄]. 상황: [한 줄]. 제약: [한 줄]. 출력: 코드만.` → Composer에 붙여넣기.

---

### 2. Replit (Replit Agent)

| 항목 | 내용 |
|------|------|
| **입력 위치** | **Ask AI** / **Agent** 패널. Agent 4(2026년 3월 기준)는 자연어로 앱·대시보드·설계까지 생성. |
| **특징** | Planning Mode로 먼저 설계 후 코드 생성, Design Canvas로 목업 확인 후 구현, 팀 협업·외부 툴(BigQuery, Slack 등) 연동. |
| **활용 팁** | 기획·설계: "역할: 기획/설계. 상황: ~. 출력: API/화면 목록 + 플로우" 로 요청. 기능 구현: "제약: 기존 Replit 프로젝트 구조 유지. 출력: 코드만" 으로 스코프 고정. |

**복사용 한 줄:**  
`[기능 구현] [기능 설명]. 역할: 풀스택. 제약: [Replit/사용 중인 스택]. 출력: 코드만.` → Ask AI/Agent에 입력.

---

### 3. Lovable (lovable.dev)

| 항목 | 내용 |
|------|------|
| **입력 위치** | 채팅 UI에서 앱/웹 요구사항을 자연어로 입력. Chat Mode Agent는 계획·디버깅·질문용(코드 수정 없음). 코드 직접 수정은 **Dev Mode**. |
| **특징** | 채팅으로 화면·플로우 설명 시 실시간 구현. Figma 임포트, Supabase DB·인증·API 자동 구성. Gemini 3 Flash 기본, Claude·OpenAI 선택 가능. |
| **활용 팁** | "역할: UI 구현. 상황: 랜딩 히어로 섹션, 다크 그라데이션. 제약: Tailwind, 모바일 우선. 출력: 해당 섹션만" 처럼 Part·제약을 명시하면 스코프가 잡힌다. 기획은 "API/화면 목록만 정리해줘" 로 요약 요청. |

**복사용 한 줄:**  
`[UI·스타일] [화면/컴포넌트]. 상황: [감각/레이아웃]. 제약: [스택/스타일]. 출력: 이 화면만 구현해줘.` → 채팅에 입력.

---

### 4. Antigravity (Google Antigravity IDE)

| 항목 | 내용 |
|------|------|
| **입력 위치** | 자연어 명령/채팅. 에디터·터미널·브라우저에 걸쳐 여러 에이전트가 동시에 작업(계획·실행·테스트·검증). |
| **특징** | Gemini 3 기반. Artifact로 워크플로·스크린샷·영상 기록. 컨텍스트 인식 완성, 실시간 협업. Claude Sonnet·GPT-OSS 등 서드파티 모델 지원. |
| **활용 팁** | "역할: 리팩터링. 상황: [파일/폴더]. 제약: 동작 변경 없이. 출력: 변경된 코드 + 요약" 처럼 **역할·상황·제약·출력**을 한 번에 주면 에이전트가 작업 범위를 잘 따른다. 테스트·디버깅은 "출력: 원인 후보 3가지 + 수정 제안" 으로 요청. |

**복사용 한 줄:**  
`[Part] [작업 내용]. 역할: [한 줄]. 상황: [대상/맥락]. 제약: [한 줄]. 출력: [코드만/요약/단계별].` → 자연어 입력창에 붙여넣기.

### 5. Claude Code (Claude.ai / API 기반 코딩)

| 항목 | 내용 |
|------|------|

| **활용 팁** | 본문 Part 1~6 예시를 그대로 복사해 사용. "기존 코드/스타일 유지" 제약을 넣으면 이미 있는 패턴을 우선 사용한다. |

## 플랫폼 선택 가이드 (2026년 3월 기준)

| 목적 | 추천 |

위 표와 부록의 **공통 원칙 + 플랫폼별 요약**만 따라도, 같은 프롬프트 구조를 모든 도구에서 재사용할 수 있다.
